# /debug — 체계적 디버깅

버그 리포트나 오류 상황에서 `/debug`를 실행하면 아래 4단계를 순서대로 수행합니다.

## 핵심 원칙

```
근본 원인 파악 없이 수정 금지
```

추측으로 패치하면 새로운 버그를 만든다. 느려 보여도 체계적 디버깅이 더 빠르다.

## 4단계 프로세스

### 1단계: 근본 원인 조사

1. **에러 메시지를 정확히 읽는다** — 스택트레이스, 라인 번호, 에러 타입 기록
2. **재현한다** — 동일한 입력/조건으로 에러를 100% 재현할 수 있는지 확인
3. **최근 변경 확인** — `git log --oneline -10`, `git diff HEAD~3` 로 원인 후보 좁히기
4. **경계 지점 조사** — FE↔BE, BE↔DB, 함수↔함수 사이 데이터 흐름 추적

수집한 증거를 정리:
```
## 조사 결과
- 에러: {에러 메시지}
- 재현: {재현 조건}
- 위치: {파일:행}
- 최근 변경: {관련 커밋}
```

### 2단계: 패턴 분석

1. **동일 패턴의 정상 코드를 찾는다** — Grep으로 유사 구현 검색
2. **정상 코드와 비교** — 어떤 차이가 있는지 식별
3. **의존성 확인** — 호출 체인, import, 상태 변수 추적

### 3단계: 가설 검증

1. **가설을 명시적으로 기술**: "X 때문에 Y가 발생한다"
2. **한 번에 하나만 변경** — 여러 수정을 동시에 하지 않는다
3. **변경 전후 결과 비교** — 같은 재현 조건으로 테스트

#### 중단 조건

아래 상황이면 **즉시 멈추고 접근법을 재검토**:

| 상황 | 조치 |
|------|------|
| 3회 이상 수정 실패 | 가설이 틀렸다 — 1단계로 복귀 |
| 수정할 때마다 새 에러 발생 | 아키텍처 문제 — 사용자에게 보고 |
| 재현이 안 됨 | 환경/타이밍 문제 — 조건 재조사 |

### 4단계: 수정 구현

1. **최소한의 변경**으로 수정 — 리팩토링을 섞지 않는다
2. **원래 재현 조건으로 검증** — 에러가 사라졌는지 확인
3. **인접 기능 확인** — 수정이 다른 곳을 깨뜨리지 않았는지 체크
4. `// [FIX]` 주석 추가 (프로젝트 컨벤션)

## 프로젝트별 주의 지점

| 영역 | 흔한 원인 |
|------|----------|
| `compare()` | 변수 섀도잉, null/undefined 체크 누락 |
| Benefits | `val`(숫자) vs `checked`(boolean) 혼동 |
| OT 계산 | 포괄임금(`inclusive`) vs 비포괄(`separate`) 분기 |
| API | `%s` 대신 f-string 사용한 SQL |
| 캐시 | 데이터 변경 후 `cache.delete()` 누락 |
| 인증 | `Depends(get_current_user)` 누락 |

## 리포트 형식

디버깅 완료 시 요약 출력:

```
## 디버깅 리포트
- 증상: {에러 설명}
- 근본 원인: {왜 발생했는지}
- 수정: {파일:행} — {변경 내용}
- 검증: {확인 방법과 결과}
```
