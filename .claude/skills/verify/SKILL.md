# /verify — 완료 전 검증 게이트

작업을 마치고 커밋/완료 선언 전에 `/verify`를 실행하면 아래 체크리스트를 수행합니다.

## 핵심 원칙

```
증거 없는 완료 주장은 거짓이다
```

"될 것 같다", "맞을 것이다"는 검증이 아니다. 명령을 실행하고 출력을 확인한 다음에만 완료를 주장한다.

## 검증 게이트

### 1단계: 변경 범위 파악

```bash
git diff --stat          # 변경된 파일 목록
git diff                 # 실제 변경 내용
```

### 2단계: 변경 유형별 검증

| 변경 유형 | 검증 방법 | "통과"의 증거 |
|----------|----------|-------------|
| **FE (index.html)** | 브라우저에서 해당 화면 확인 | 렌더링 정상, 콘솔 에러 없음 |
| **BE (API)** | `curl` 또는 API 클라이언트로 엔드포인트 호출 | 기대한 응답, status 200 |
| **DB (schema.sql)** | DDL 문법 확인, 컬럼 COMMENT 존재 | 문법 오류 없음 |
| **CSS** | 반응형 확인 (480px 브레이크포인트) | 레이아웃 깨지지 않음 |
| **SQL 쿼리** | `%s` 플레이스홀더 사용 확인 | f-string 없음 |

### 3단계: 컨벤션 체크

- [ ] FE 변수명 camelCase, 함수명 camelCase
- [ ] BE 함수명 snake_case, 클래스 PascalCase
- [ ] SQL에 f-string 없음 (`%s` 사용)
- [ ] DB 컬럼에 한국어 COMMENT 존재
- [ ] ENUM 미사용 (VARCHAR + COMMENT)
- [ ] UI 텍스트 한국어
- [ ] `innerHTML`에 사용자 입력 시 `esc()` 사용

### 4단계: 인접 영향 확인

변경한 함수/엔드포인트를 **호출하는 쪽**을 Grep으로 확인:

```
변경한 함수명 → Grep으로 호출처 검색 → 시그니처 변경이 호출처와 호환되는지 확인
```

### 5단계: 결과 리포트

```
## 검증 리포트
- 변경 파일: {파일 목록}
- 검증 항목:
  - ✅ {검증한 항목}: {증거}
  - ✅ {검증한 항목}: {증거}
  - ⚠️ {확인 불가 항목}: {이유} (서버 미실행 등)
- 컨벤션: ✅ 통과 / ⚠️ {위반 사항}
- 인접 영향: ✅ 없음 / ⚠️ {영향받는 코드}
```

## 자동 트리거 조건

이 검증은 다음 상황에서 **자동으로 수행해야** 합니다:

1. 커밋 직전
2. "완료", "끝", "다 했어" 등 완료 의사 표현 시
3. PR 생성 직전
4. 다음 작업으로 넘어가기 전

## 위험 신호 — 즉시 멈추기

아래 표현을 쓰려는 자신을 발견하면 검증이 빠진 것:

- "아마 될 겁니다"
- "문제없을 것 같습니다"
- "수정했으니 괜찮습니다"
- "이전에 확인했습니다" (이번 변경 후 재확인 필요)

→ **멈추고, 명령을 실행하고, 출력을 확인한 다음** 주장하세요.
