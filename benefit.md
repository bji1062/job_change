# 복지 카테고리 분석 & 유사도 매핑 리포트

## 목적

사용자가 공유한 회사별 복지 원문 데이터(소위 7개 카테고리 체계, 약 90여 개 회사)를 분석하여

1. **현재 7개 카테고리 체계가 적절한지** 평가하고
2. **이직 시 회사 간 복지를 비교할 때 사용할 유사도(동일/유사/유일) 매핑 규칙**을 설계한다.

이 리포트는 코드를 변경하지 않는 문서 산출물이며, 향후 `js/data.js`·`server/seed/*.sql`·`compare.js` 등의 리팩토링 근거로 사용된다.

### 1.5 주요 관찰

1. **레이블 편중(라벨 인플레이션)**: 원본은 36개 헤더를 쓰지만 실질 의미는 더 적다. 예) `통근/교통비`·`주차비/주차장`·`출퇴근/근무제`·`업무효율지원(기숙사)` 사이에 경계가 모호.
2. **시간휴가 대분류 과소 점유**: 연차·휴가·리프레시·장기근속 관련 항목이 매우 많은데도 대분류로는 2개 서브만 귀속. `휴양시설`·`집중휴가(하계/동계)`를 시간휴가로 재귀속하면 더 균형 잡힘.
3. **보상금전 대분류 과다 점유**: 연봉·인센티브·스톡·통근·식대·통신·자사할인까지 몰려 있다. 이 중 **통근/식대/자사할인**은 보상이라기보단 "근무환경 부속 보상"으로 분리 후보.
4. **경계 모호 서브카테고리(다중 귀속 필요)**:
   - `장기근속포상` → 시간휴가(휴가 부분) + 보상금전(포상금 부분)
   - `자기계발/복지카드` → 성장/커리어(학자금·교육) + 라이프스타일(복지포인트 자유 사용)
   - `휴양시설` → 시간휴가(집중휴가와 묶임) + 라이프스타일(숙박·여가)
   - `경조사지원` → 가족(휴가·관계) + 보상금전(경조금)
   - `식대/점심제도` → 보상금전(식대 현금) + 근무환경(사내식당)
5. **결측 빈번 카테고리**: `포괄/비포괄`·`스톡옵션`·`RSU`·`통신비`·`주차비`·`회식/회식문화`는 대부분 회사에서 미기재 → 누락을 "없음"과 "명시안함"으로 구분할 방법 필요.
6. **하이라이트 섹션**: 일부 회사(네오위즈·네패스·카카오게임즈·엔씨소프트·펄어비스·카카오페이·카카오VX·파크시스템스·KLT·네이버파이낸셜·SK텔레콤·엔카닷컴·하이브로·더블유게임즈)에는 `하이라이트` 요약 문구가 있음 → 동일 복지도 "강조 여부"로 무게가 달라질 수 있음.
7. **금액·조건·수치 정보의 불규칙성**: "복지포인트 200만원", "연 250만원", "월 20만원" 같이 단위/기간이 뒤섞여 있음 → 정규화 필수.

### 1.6 Phase 1 요약

- 원본은 **36개 서브카테고리** × **93개 회사** 규모.
- 사용자의 **7대분류 체계**는 상위 체계로 성립 가능하나, **시간휴가 축소·보상금전 팽창·다중 귀속 모호성** 세 가지 문제가 드러남.
- Phase 2에서는 이 36개 서브카테고리의 실제 항목을 정규화하고 동의어 그룹(canonical key)을 정의한다.
- Phase 3에서 7대분류 재설계안(유지/분할/병합)을 제시한다.

## Phase 2 — Normalization & Synonym Dictionary (정규화 & 동의어 사전)

### 2.1 정규화 규칙 (Normalization Rules)

원본 복지 문구는 금액·조건·설명이 섞여 있어 그대로는 비교할 수 없다. 다음 규칙을 순서대로 적용한다.

1. **공백/구두점 정리**: 좌우 공백 제거, 연속 공백 단일화, 괄호 주석 `(본인+가족)`·`(월 18,000원 × 240일)` 분리 보관.
2. **단위 숫자 추출**: `월 OO만원`, `연 OO만원`, `주 OO일`, `일 OO원`, `N년` 등 수치 토큰을 `amount / period / condition` 필드로 분리.
3. **한자·영문 병기 통일**: `Refresh` ↔ `리프레시`, `OJT` ↔ `신입교육`, `MBA` ↔ `경영대학원` 등 동일 의미 문자열 통일.
4. **강도 표현 제거**: "다양한", "충분한", "자유로운", "최고 수준의", "다채로운" 등 과장형 형용사는 매칭에서 제외(참고 필드로만 보존).
5. **핵심 명사 추출**: "구내식당 (아침·점심·저녁 100%)" → 키워드 `{구내식당, 삼시세끼}`; "법인카드 식사 (점심+저녁 100%)" → `{법인카드, 점심, 저녁}`.
6. **긍정/부정 필터**: "회식 강요 없음", "야근 강요 없음" 등 **부재/금지**를 표현하는 문장은 역 플래그(`anti-*`)로 별도 저장.
7. **대상 범위 태그**: `본인` / `배우자` / `자녀` / `부모` / `직계가족` / `배우자부모` 범위를 `scope[]` 배열로 보존(예: 건강검진 scope=[본인,배우자,부모]).
8. **수량 단위 정규화**:
   - 금액: `만원` → `10000₩`, `천만원` → `10000000₩`, `억원` → `100000000₩`
   - 기간: `연` → `year`, `월` → `month`, `주` → `week`, `일` → `day`
   - 예: "연간 복지포인트 200만원" → `{kind:'welfare_point', amount:2000000, period:'year'}`

### 2.2 Canonical Key 체계

복지 문구의 의미 단위를 `canonical_key`로 묶는다. `canonical_key`는 **소분류 수준**(서브카테고리보다 더 세밀)으로 정의되며, 서브카테고리 내부의 서로 다른 표현들을 하나로 수렴시킨다.

형식: `<domain>.<concept>[.<variant>]`

### 2.4 비동의어(Anti-pattern) 예시

매칭 시 걸려들면 안 되는 흔한 함정:

- `"복지포인트"` vs `"복지카드"` — 이름은 비슷하지만 한쪽은 포인트 시스템, 한쪽은 체크카드. 실제 용처가 같으면 `comp.welfare_point`로 동일 취급.
- `"식대 지원 월 15만"` vs `"사내식당 무료"` — 제공 형태가 달라 `동일`이 아닌 `유사`로만 매칭(카테고리 이웃).
- `"재택근무"` vs `"원격근무"` — 동일 `leave.remote`지만 빈도(주 1 vs 주 5)에 따라 `동일/유사` 분리 필요.
- `"연봉 협상 연 1회"`는 `comp.salary_review`로 별도; `"성과급"`과 혼동 금지.
- `"경조사 지원"`은 현금(경조금) + 휴가(경조휴가) + 화환 3요소가 결합된 **복합 항목** — 동일성 판정 시 3요소를 분해해 비교.

### 2.6 Phase 2 요약

- **정규화 8단계 규칙**과 **9개 도메인 × 50+개 canonical_key** 동의어 사전 초안 완성.
- 모든 canonical_key는 `<domain>.<concept>[.<variant>]` 규칙을 따르며, Phase 4의 유사도 판정에 직접 사용된다.
- 동의어 사전은 코드로 옮기기 쉽도록 YAML 형태를 유지했다(필요 시 JSON 변환 용이).
- Phase 3에서는 이 canonical_key 분포를 바탕으로 7대분류를 재설계한다.

## Phase 3 — Category Review (카테고리 재설계)

### 3.1 평가 기준

각 대분류에 대해 다음 세 가지 관점으로 `유지(Keep) / 분할(Split) / 병합(Merge) / 재정의(Redefine)` 판정을 내린다.

1. **의미 응집성**: 대분류 내부 항목들이 "한 문장으로 설명 가능한 성격"을 공유하는가?
2. **비교 유용성**: 이직 시 회사 간 비교에서 대분류 단위 합산이 의미를 만드는가?
3. **결측 내성**: 회사마다 누락이 잦은 항목이 한 대분류에 몰려 있으면 비교가 왜곡되는가?

### 3.2 현행 7대분류 진단

| 대분류 | 대표 내용 | 응집성 | 비교 유용성 | 결측 내성 | 판정 |
|---|---|---|---|---|---|
| 보상금전 | 연봉·인센티브·스톡·식대·통근비·통신비·자사할인·복지포인트 | 낮음 (이질적) | 높음 | 중 | **분할** |
| 근무환경 | 포괄/비포괄·재택·유연근무·주차·휴게시설·업무장비·회식 | 중 | 중 | 낮음 (누락 많음) | **분할** |
| 건강의료 | 검진·의료비·단체보험·사내의원·심리상담·헬스장 | 높음 | 높음 | 높음 | **유지** |
| 시간휴가 | 연차·리프레시·장기근속휴가·집중휴가·출산/육아휴직 | 높음 | 높음 | 중 | **재정의** (범위 확장) |
| 성장/커리어 | 교육·MBA·어학·도서·자기계발비 | 높음 | 중 | 중 | **유지** |
| 가족 | 임신/출산·어린이집·자녀학자금·경조사 | 중 | 중 | 높음 | **유지** (경계 명확화) |
| 라이프스타일 | 생일/기념일·휴양시설·동호회·카페스낵·자사제품·기타 | 낮음 (잡동사니) | 낮음 | 낮음 | **분할** |

### 3.3 권장안 A — **보수적 재설계 (7 → 8 분류)**

현행 골격을 유지하면서 **가장 이질적인 "라이프스타일"을 둘로 쪼개는** 최소 변경안. 시간휴가 대분류는 항목 범위만 재정의.

```
[1] 보상·금전            (현행 유지)
[2] 근무유연성            (신설: 재택/유연/출퇴근/연차/휴가제도/리프레시/집중휴가)
[3] 근무환경·장비         (신설: 장비/사무공간/주차/기숙사/업무효율/회식/식당시설)
[4] 건강·의료             (현행 유지)
[5] 성장·커리어           (현행 유지, 도서비 포함)
[6] 가족·돌봄             (현행 가족 + 출산/육아휴직 이관, 경조사 포함)
[7] 여가·생활             (휴양시설/동호회/자사할인/생일기념/카페스낵 등)
[8] 편의·부가             (수면실·마사지·웰컴키트·상비약 등 "기타사항")
```

장점: 라이프스타일 잡동사니를 정리. 단점: 여전히 보상금전이 팽창.

### 3.4 권장안 B — **적극적 재설계 (7 → 9 분류)** 🌟 권장

이질성 해소와 비교 유용성 최적화를 모두 고려한 안. **사용자의 원래 7분류를 최대한 보존하되**, 이질적 대분류는 분할하고, 결측률이 높은 핫 스팟 `근무유연성`을 별도 축으로 승격.

```
[1] 보상·금전 (Compensation)
    - 연봉/인센티브/성과급(PS·PI)/스톡옵션·RSU/명절상여/우수사원 포상금
    - ※ 통근/식대/통신비/자사할인은 [9] 경제적 혜택으로 분리

[2] 근무유연성 (Work Flexibility) 🌟 신설
    - 포괄/비포괄/재택/원격/시차출퇴근/유연근무/PC-OFF/패밀리데이/거점오피스
    - ※ "지금·이 시간을 자유롭게 쓰는가"가 핵심

[3] 근무환경 (Work Environment)
    - 사무공간/장비(맥북·허먼밀러)/기숙사/사택/사내식당 시설/회식 문화/수면실/라운지

[4] 시간·휴가 (Time Off)
    - 법정연차/하계집중휴가/리프레시/안식월/장기근속휴가/창의휴가/보건휴가/가족돌봄휴가
    - ※ 출산/육아 휴직은 [6]으로 이관 (가족 축과 묶음)

[5] 건강·의료 (Health)
    - 종합검진/단체보험/사내의원·약국/심리상담/헬스장·운동비/독감접종

[6] 가족·돌봄 (Family & Care)
    - 임신/출산/육아휴직/어린이집/자녀학자금/자녀 입학선물/가족 의료비·건강검진/경조사

[7] 성장·커리어 (Growth)
    - 직무교육/리더십/어학/MBA·석박사 파견/지역 전문가/자기계발비/도서비/사내공모

[8] 여가·라이프 (Leisure)
    - 휴양시설·콘도/동호회·소모임/생일기념/웰컴키트/사내 편의시설(안마·북카페)/기타

[9] 경제적 부가혜택 (Economic Perks) 🌟 신설
    - 식대/통근비/주차비/통신비/사내대출(이자 지원)/복지포인트/자사·계열 할인/상품권
    - ※ 개별 금액은 작지만 합산 시 보상금전과 맞먹는 "숨은 연봉"
```

**권장안 B를 선택한 이유:**

1. **보상금전의 3층화**:
   - [1] 보상·금전 = 진짜 현금 보상(연봉·PS·스톡)
   - [9] 경제적 부가혜택 = 실질적으로 지갑에 들어오는 현금등가 혜택(식대·통근·복지포인트)
   - 두 축을 분리해야 "명목 연봉은 낮아도 실질 보상은 높다"는 이직 비교의 핵심 인사이트를 낼 수 있음.
2. **근무유연성을 독립 축으로**: 데이터에서 압도적으로 빈번(약 80% 이상 회사에서 언급)하고, 최근 이직자의 의사결정에서 최상위 우선순위이므로 단독 축이 합당.
3. **시간·휴가 축 재정의**: 기존 "시간휴가"가 너무 좁아서 과소 점유였는데, 집중휴가·안식월·보건휴가까지 묶으면 9개 중 중위권 크기로 균형.
4. **출산/육아 휴직을 가족 축으로 이관**: 동일 사건(임신·출산)을 중심으로 묶는 게 사용자 멘탈모델과 맞음.
5. **라이프스타일 잡동사니 해소**: 여가·편의·기타를 [8] 여가·라이프 하나로 축소(권장안 A의 [8] 편의·부가는 다시 합침 — 너무 잘게 쪼개면 결측이 늘어 비교 가치 감소).

### 3.6 다중 귀속(Multi-bucket Tagging)의 필요성

원본 데이터에서 **6개 서브카테고리**가 둘 이상의 대분류에 걸쳐 있다. 단일 귀속으로 강제하면 "어느 대분류에도 억지로 끼워넣는" 손실이 생긴다. 권장안 B는 이를 **다중 태깅(multi-bucket)** 으로 해결한다:

- 한 복지 항목이 최대 2개(예외적으로 3개)의 대분류 태그를 가질 수 있다.
- 합산 시 "주 카테고리"에만 금액을 귀속시키고, 보조 카테고리는 표시용 태그로만 사용.
- 예: `자녀 학자금 지원 300만원` → 주=가족·돌봄, 보조=경제적 부가혜택(실질가치 300만).

이 방식은 "중복 계상"을 막으면서도 대분류별 합계가 사용자의 멘탈모델과 일치하게 한다.

### 3.7 간단 비교표

| 항목 | 현행 7분류 | 권장안 B (9분류) |
|---|---|---|
| 분류 수 | 7 | 9 (+2) |
| 가장 큰 축 | 보상금전 (과도 집중) | 보상·금전 + 경제적 부가 (분리) |
| 가장 작은 축 | 시간휴가 (2개 서브) | 시간·휴가 (5+ 서브, 재정의) |
| 근무유연성 위치 | 근무환경에 묻힘 | 독립 축 |
| 다중 태깅 | 없음 | 있음 (6개 서브) |
| 코드 변경 영향 | — | `cat` 문자열 확장 + 다중 태그 배열화 |

### 3.8 Phase 3 요약

- **현행 7분류의 3가지 문제**(보상금전 팽창·시간휴가 축소·라이프스타일 잡동사니)를 확인.
- **권장안 B(9분류)**: 근무유연성·경제적 부가혜택을 신설하고 시간·휴가 축을 재정의.
- 6개 서브카테고리는 **다중 태깅**으로 처리하여 의미 손실 없이 합산 가능.
- Phase 4에서는 이 분류 체계 위에서 `동일/유사/유일` 판정 규칙을 설계한다.

## Phase 4 — Similarity Mapping Rules (동일 / 유사 / 유일 판정)

### 4.1 판정 3단계 정의

두 회사 A, B의 복지 항목을 짝지어 세 라벨 중 하나를 부여한다.

| 라벨 | 한글 | 조건 (직관) | 비교 결과 UI |
|---|---|---|---|
| **identical** | 동일 | 같은 의미 + 같은 수치/조건 | 양측 동일 표시, 차이 없음 |
| **similar** | 유사 | 같은 의미지만 수치/조건/범위가 다름, 또는 동일 카테고리 이웃 항목 | 양측 비교 표시, 차이 강조 |
| **unique** | 유일 | 한쪽에만 존재하고 상대 쪽에 이웃 항목도 없음 | 한쪽만 표시, "상대 없음" 라벨 |

### 4.2 매칭 파이프라인

```
┌─────────────────────────────┐
│  A의 benefit_a             │
│  B의 benefit_b             │
└──────────────┬──────────────┘
               ▼
    1) canonicalKey(a), canonicalKey(b)
               │
               ▼
    ┌──────────────────────────┐
    │  key_a == key_b ?        │
    └──┬───────────────────┬───┘
       │ YES               │ NO
       ▼                   ▼
   sameValue(a,b) ?    sameCategory &
       │               inNeighborSet ?
   ┌───┴────┐          ┌───┴────┐
   YES      NO         YES      NO
   │        │          │        │
   ▼        ▼          ▼        ▼
 동일     유사        유사      유일
```

### 4.3 의사코드 (Pseudocode)

```javascript
function classify(a, b) {
  const ka = canonicalKey(a);
  const kb = canonicalKey(b);

  // 1) 완전 동일
  if (ka === kb) {
    if (sameValue(a, b) && sameScope(a, b) && sameCondition(a, b)) {
      return 'identical';  // 동일
    }
    return 'similar';      // 키는 같지만 수치·조건이 다름
  }

  // 2) 이웃 집합 (동일 카테고리 내 인접 의미)
  if (sameCategory(a, b) && inNeighborSet(ka, kb)) {
    return 'similar';      // 유사
  }

  // 3) 그 외 — 상대 쪽에 매칭 없음
  return 'unique';         // 유일
}
```

보조 함수 정의:

- `canonicalKey(x)`: Phase 2 동의어 사전 역인덱스에서 조회. 미등록 문자열은 `env.raw:<해시>`로 폴백.
- `sameValue(a,b)`:
  - 금액: `abs(a.amount - b.amount) / max(a.amount, b.amount) ≤ 0.10` (±10%)
  - 기간: 주기 단위 일치(`year==year`)
  - 빈도/횟수: 정수 일치 (연 1회 ↔ 연 1회)
  - 한쪽 금액 결측 + 한쪽 금액 명시: **false** (부분 정보만으로 동일 판정 금지)
- `sameScope(a,b)`: `scope` 배열 집합 동등(`{self,spouse,child} == {self,spouse,child}`). 부분집합 관계는 `similar`로 떨어뜨림.
- `sameCondition(a,b)`:
  - 근속 요건(`근속 3년+` vs `근속 5년+`) 비교.
  - 직급/직군 한정(`팀장 이상` vs `전 직원`) 비교.
  - 한쪽만 조건 있고 다른 쪽은 "전원"이면 **false**.
- `sameCategory(a,b)`: Phase 3 권장안 B의 9개 대분류 기준.
- `inNeighborSet(ka,kb)`: 아래 4.4의 이웃 집합에 포함되는지 확인.

### 4.5 태이-브레이커 (Tie-breaker) 규칙

여러 후보가 경쟁하는 경우(한쪽에 같은 canonical_key가 여러 개 있는 등):

1. **1:1 우선 매칭**: 양측 모두에서 같은 `canonical_key`가 정확히 하나면 자동 매칭.
2. **N:1 분산 매칭**: A쪽에 두 항목, B쪽에 한 항목이면 → A의 두 항목 중 금액이 더 가까운 쪽이 `similar`, 나머지는 `unique`.
3. **금액 없음 vs 있음**: 한쪽은 "있다"만 명시, 한쪽은 수치 있으면 → `similar`로 고정(동일 아님).
4. **조건부 항목**(예: "자녀 있는 경우만"): `checked=false` 또는 `conditional=true` 플래그가 한쪽만 붙으면 → `similar`.
5. **정성적 항목**(qualText, 금액=0): 금액 비교 스킵하고 텍스트 임베딩/키워드 Jaccard로 비교(임계 0.6 이상이면 `similar`).

### 4.7 결측 처리 규칙

한쪽 데이터에 어떤 항목이 **아예 등장하지 않을 때**:

1. **카테고리 레벨 결측**: A에 `family.childcare_onsite`가 있는데 B에는 `family.*` 전체가 없음 → `unique`.
2. **카테고리 레벨 매칭**: A에 `family.childcare_onsite`, B에 `family.childcare_subsidy`(보육비)만 있음 → `similar` (이웃 집합 해당).
3. **명시 없음 vs 공식 부재 구별 불가**: 본 분석에서는 "원문에 없으면 없다"로 간주. 단 UI에 "데이터 불명"과 "없음"을 구별하는 배지 추가 권장(Phase 5 제안).
4. **qualText만 있는 항목**(금액 0): scope/키워드로만 비교. 한쪽에 유사 표현 없으면 `unique`.

### 4.8 우선순위 및 가중치 힌트 (선택적)

이직 비교에서 "같은 유사 등급이라도 체감 중요도"가 다르다. 다음은 선택 가중치(참고용, Phase 5에서 정식 채택은 후속 작업):

| 근무유연성 | 0.7 | 최근 이직 핵심 축 |

### 4.9 Phase 4 요약

- **판정 3단계**(동일/유사/유일)와 **4-step 매칭 파이프라인** 정의.
- **이웃 집합**(neighbor sets) 9개 도메인 × 20+ 그룹을 Phase 2 동의어 사전 위에 구축.

- 이 규칙은 `classify(a, b) → 'identical'|'similar'|'unique'` 함수로 곧바로 구현 가능.

### 5.5 리포트 외 권장 후속 분석

본 리포트 범위를 벗어나지만 이어서 해볼 만한 작업:

1. **빈도 통계**: 93개 회사에 걸쳐 각 canonical_key의 등장 횟수를 집계하면 "시장 표준 복지"를 식별할 수 있음. 90% 이상 회사에 있는 키는 `baseline`, 10% 미만은 `rare`로 태깅하여 비교 시 차별점 강조.
2. **업종·기업유형별 프로파일**: `large / startup / mid / foreign` 유형별로 대분류 평균 스코어 산출 → 이직 시 "유형 전환"이 가져올 복지 변화를 수치화.
3. **금액 환산 규칙**: 정성적 항목("사내 카페 무료", "어린이집 운영")의 시장 가격 환산값을 가이드로 부록 추가 → 총액 비교의 빈 칸 메우기.
4. **사용자 가중치 커스터마이즈**: §4.8 가중치를 프로파일러 결과(`DIMS`)와 연동해 개인화(`자녀 있음 → family 1.5x`).
5. **데이터 품질 UI**: Phase 4.7에서 언급한 "명시 없음 vs 공식 부재 구별" 배지 도입 → 사용자가 "데이터가 불충분해서 0"인지 "실제로 없어서 0"인지 구별 가능.
